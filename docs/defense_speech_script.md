## 一、开场

各位老师好，我汇报的项目是《量化回测平台设计与实现》。

这个项目主要围绕虚拟货币现货研究流程展开，完成了实时行情采集、策略回测、异步任务管理、风险指标展示和智能分析几个模块。下面我会按照项目背景、技术栈与架构、功能实现、关键难点、验证结果和总结展望的顺序进行介绍。

## 二、逐页讲稿

### 第 1 页：封面与汇报结构

这一页是项目整体介绍。

本项目实现的是一个基于 Vue 3、FastAPI 和 Python 量化引擎的回测平台。核心功能包括实时行情、策略回测、异步任务和智能分析。右侧是当前演示数据库中的基本数据，包括 K 线记录、回测记录和分析记录。

本次答辩的重点是说明平台如何把行情数据、回测任务和分析结果组织成一个可以运行、可以复查、可以演示的工程系统。

### 第 2 页：项目背景与建设目标

项目背景来自个人量化研究的基本流程。一般需要先获取行情数据，再设计策略，然后进行历史回测，最后根据收益、回撤和交易记录评估策略。

如果只用脚本完成这些工作，容易出现几个问题：数据来源不统一，回测参数不容易复现，结果不方便保存，也不方便在页面上展示给非技术用户看。

所以本项目的阶段目标是完成一个课程演示平台，能够支持行情观察、策略回测、结果分析和演示复现。当前不涉及真实下单，也不作为实盘交易系统使用。

### 第 3 页：需求范围与完成情况

这一页说明当前做到了哪些需求。

第一，平台已经完成注册登录、用户数据隔离、策略管理和演示脚本。

第二，行情模块支持从 Binance 拉取 K 线，写入数据库，并在前端展示数据健康状态。

第三，回测模块支持内置策略和自定义策略，回测任务通过异步 job 执行，支持状态查询、取消、重试和失败恢复。

第四，分析模块支持 Gemini、OpenAI 和本地引擎切换，分析结果会保存到数据库。

范围上，本期主要支持加密货币现货多头，标的以 BTCUSDT 为主，周期覆盖 1m、1h 和 1d。

### 第 4 页：平台技术栈

平台采用前后端分离结构。

前端使用 Vue 3、Vite 和 ECharts，负责登录页面、行情图表、回测配置表单和结果展示。

后端使用 FastAPI、Pydantic 和 JWT，负责接口、参数校验、鉴权和业务编排。

数据层使用 SQLAlchemy、Alembic 和 SQLite。SQLite 适合本地课程演示，Alembic 用于维护数据库迁移。

外部服务主要包括 Binance REST 行情接口，以及 Gemini 和 OpenAI 模型接口。

量化计算部分封装在 `quant_engine` 包中，包括 strategy、datafeed、broker、analyzer 和 models 几个模块。

### 第 5 页：系统总体架构

这一页是系统架构。

最上层是用户界面，包含登录、行情观察、策略管理、回测配置、结果展示和分析页面。

第二层是 FastAPI 服务，负责鉴权、参数校验、任务创建、数据查询和分析接口。

第三层是业务服务层，主要包括 `market_data_service`、`backtest_service`、`backtest_job_service` 和 `ai_analysis_service`。

底层是量化核心和数据库。`quant_engine` 负责策略执行、撮合、风控和指标计算；SQLite 保存用户、策略、K 线、任务、回测结果和分析结果。

这个架构的好处是前端、业务服务和量化核心职责分开，后续如果要替换数据库、增加 worker 或扩展数据源，不需要重写整个系统。

### 第 6 页：功能模块实现

这一页从用户使用流程来说明功能。

用户先注册或登录，登录后可以进入平台页面。然后在行情页面查看 BTCUSDT 的不同周期 K 线，也可以手动同步最新行情。

接着在策略管理页面选择内置策略，或者从模板复制出自定义策略。回测页面可以配置标的、周期、时间范围、初始资金和策略参数。

提交回测后，任务中心会展示 job 状态。回测成功后可以打开结果页，查看 K 线成交点、权益曲线、指标和交易明细。最后可以对该回测结果触发智能分析，查看评分、风险提示、执行计划和建议参数。

### 第 7 页：实时行情数据链路

这一页介绍行情数据如何进入系统。

后端会先根据 symbol、timeframe 和 range 计算请求窗口，然后调用 Binance K 线接口分页拉取数据。由于接口单次返回数量有限，所以代码中会按时间继续请求后续数据。

拉到的数据会统一转换为 `open_time`、`close_time`、open、high、low、close、volume 等字段，再按 symbol、timeframe 和 open_time 做 Upsert，避免重复插入。

入库后，系统会计算数据健康状态，包括最新 K 线距离当前时间的 age、时间序列中 gap 的数量，以及缺失 K 线数量。

这里的关键点是：前端图表和回测服务使用的是同一份持久化行情数据。如果数据库覆盖了回测区间，回测会优先使用数据库 K 线；如果覆盖不足，再回退到 CSV 样例数据。

### 第 8 页：异步回测任务链路

这一页是项目里的一个实现重点。

前端提交回测参数后，后端先创建一条 `BacktestJob` 记录，初始状态是 queued。

后端用 `ThreadPoolExecutor` 提交后台任务。任务开始执行后，状态变为 running。后台任务会调用 `run_backtest()`，并使用独立数据库 Session 执行回测和保存结果。

回测成功后，系统把结果写入 `backtest_records`，并在 job 上记录 `result_backtest_id`。前端通过轮询 job 状态展示进度，成功后再自动打开回测结果。

任务状态包括 queued、running、succeeded、failed 和 cancelled。系统还支持取消任务、失败后重试，以及服务启动时把中断任务恢复为失败状态，避免任务卡在 running。

### 第 9 页：回测引擎执行模型

这一页介绍量化引擎内部的执行顺序。

`BacktestEngine.run()` 是一个逐 Bar 循环。每处理一根 K 线，首先执行上一轮策略提交的待成交订单，成交价使用当前 Bar 的 open。

然后系统把当前 K 线加入 `StrategyContext.history`，策略只能基于当前及历史数据计算信号。接着调用 `strategy.on_bar(kline)`，策略在这里生成买入或卖出订单。

最后 broker 按当前 Bar 的 close 更新权益曲线。交易记录、现金、持仓和手续费都由 broker 统一维护，收益率、最大回撤、Sharpe 和胜率由 analyzer 计算。

这个执行顺序可以清楚地区分信号生成、订单成交和权益更新，也方便写测试验证结果。

### 第 10 页：智能推荐分析模块

这一页介绍分析模块。

分析模块的输入采用回测记录中的结构化信息，包括策略名称、标的、周期、时间范围、初始资金、参数、收益率、最大回撤、Sharpe、交易次数和权益曲线。

Provider Router 会根据前端选择调用 Gemini、OpenAI 或本地引擎。如果外部接口失败，系统会保存 fallback reason，并返回本地分析结果。

输出结果采用固定字段，包括 score、confidence、strengths、warnings、recommendations、execution_plan 和 suggested_params。这样前端可以稳定展示，数据库也可以保存历史分析记录。

演示时可以重点展示 provider 字段，说明当前分析来自 Gemini、OpenAI 还是本地引擎。

### 第 11 页：验证结果与演示准备

这一页说明项目如何验证。

演示启动可以运行 `./scripts/restart_demo.sh`，它会先释放旧端口，然后重新启动后端和前端，再执行自检。默认后端是 8000，前端是 5173。

当前演示数据库里有 K 线记录、回测记录、分析记录和异步任务记录。后端自动化测试覆盖了用户认证、策略隔离、回测服务、异步任务、行情服务和分析服务。

录视频时建议保留一条历史成功回测和一条历史分析记录，这样即使现场外部网络波动，也能展示完整结果。

### 第 12 页：总结与展望

最后总结一下。

本项目已经完成了一个可演示的量化回测平台，包括前后端页面、实时行情同步、异步回测任务、量化回测引擎和智能分析模块。

技术收获主要包括：数据链路与计算链路解耦、任务状态持久化、回测引擎模块化，以及模型分析结果可追溯。

后续可以继续完善四个方向：第一是增加样本外和走步验证；第二是为自定义策略增加沙箱隔离；第三是把进程内线程池替换成独立 Worker 和队列；第四是扩展更多市场和策略类型。

我的汇报到这里，谢谢各位老师。

## 四、老师提问预案

### 1. 为什么选择虚拟货币，不做 A 股或美股？

可以回答：

本期是课程项目，优先保证核心链路完整。虚拟货币公开 K 线接口比较容易获取，适合演示实时同步和回测。系统结构里保留了 asset_class、market_type 和 position_mode 字段，后续可以扩展到更多市场，但本期不把美股和期货双向作为交付范围。

### 2. 现在的行情是否实时？

可以回答：

行情来自 Binance REST K 线接口，前端支持手动同步和定时刷新。数据同步后会写入 `market_klines` 表，并计算最新 K 线时间、age、gap 和 missing bars。演示中默认显示数据库里的最新数据；如果现场网络可用，可以手动同步刷新。

### 3. 为什么访问 5174 不行？

可以回答：

项目默认前端端口是 `5173`，后端端口是 `8000`。如果需要使用 5174，可以运行：

```bash
FRONTEND_PORT=5174 ./scripts/restart_demo.sh
```

默认录屏和答辩建议使用 `http://127.0.0.1:5173`。

### 4. 回测怎么避免未来数据？

可以回答：

回测引擎按逐 Bar 顺序执行。当前 Bar 开始时只执行上一轮产生的 pending order，成交价用当前 Bar 的 open；之后才把当前 K 线加入历史，并让策略生成下一轮信号；最后用 close 更新权益。这样信号生成和成交之间有明确顺序，不会用同一根 K 线的收盘信息再按同一收盘价成交。

### 5. 回测撮合是否考虑手续费？

可以回答：

考虑了。`SimulatedSpotBroker` 里有 `fee_rate`，默认是 0.001。买入时扣除 `gross_cost + fee`，卖出时收到 `gross_value - fee`。交易记录里也保存了 fee 字段。

### 6. 支持做空或期货双向吗？

可以回答：

当前版本不支持。当前 broker 是现货多头模型，主要用于课程演示。后续如果扩展期货，需要增加保证金、杠杆、资金费率、强平规则和 long/short position 管理。

### 7. 为什么要做异步回测？

可以回答：

回测可能需要读取较长时间区间的数据，或者未来执行更复杂策略。如果直接在 HTTP 请求里同步计算，页面会长时间等待，也不方便取消和重试。异步 job 可以把提交、执行、状态查询和结果展示拆开，更适合工程化扩展。

### 8. 当前异步任务有什么局限？

可以回答：

当前使用进程内 `ThreadPoolExecutor`，适合本地演示和单实例运行。生产环境应该换成独立队列和 Worker，比如 Redis Queue、Celery 或其他任务系统，并增加心跳、超时和资源限制。

### 9. 自定义策略安全吗？

可以回答：

当前自定义策略适合本地课程演示。系统会在保存前校验 `generate_signals(klines, params)` 是否存在，但还没有完整沙箱。正式开放给用户前，需要把策略执行放到隔离进程，限制文件访问、网络访问、执行时间和内存。

### 10. 智能分析是真调用模型还是本地规则？

可以回答：

系统支持三种模式：Gemini、OpenAI 和本地引擎。页面会展示实际 provider。如果配置了 Gemini key 并且网络可用，会调用 Gemini；如果外部 API 失败，会显示 fallback reason 并回退本地分析，不会隐藏来源。

### 11. 模型分析的输入是什么？

可以回答：

输入来自回测记录，包括策略名称、标的、周期、时间范围、初始资金、策略参数、收益率、最大回撤、Sharpe、胜率、交易次数、权益曲线和交易记录。输出是固定字段，方便前端展示和数据库保存。

### 12. 模型会不会直接给买卖建议？

可以回答：

当前定位是回测结果分析和参数实验建议，不直接作为实盘买卖信号。它会给出风险等级、警告、适用场景、执行计划和建议参数，最终仍需要通过回测和样本外验证确认。

### 13. 数据库为什么用 SQLite？

可以回答：

SQLite 适合课程本地演示，部署简单，不依赖额外数据库服务。项目使用 SQLAlchemy 和 Alembic 维护模型和迁移，后续可以切换到 MySQL 或 PostgreSQL。

### 14. 如何证明工程可复现？

可以回答：

项目提供 `restart_demo.sh` 一键重启脚本、`check_demo.sh` 自检脚本、Alembic 迁移和自动化测试。当前自检会检查后端、前端、数据库、K 线数量和 Gemini 配置。

### 15. 如果现场 Gemini 或 Binance 访问失败怎么办？

可以回答：

Gemini 失败时可以切换本地引擎，也可以打开历史分析记录。Binance 失败时可以展示数据库中已经保存的近期 K 线，回测覆盖不足时还有 CSV 样例数据作为回退。

### 16. 项目最大的不足是什么？

可以回答：

主要不足有三个：自定义策略还没有沙箱隔离；异步任务还在 API 进程内线程池中；行情定时同步目前主要由前端触发。后续会把这些迁移到独立 Worker、服务端调度器和更完整的安全隔离中。

### 17. 这个项目的核心工作量在哪里？

可以回答：

核心工作量不只在页面，而是在几条链路的打通：行情同步和入库、回测任务异步执行、量化引擎撮合和指标计算、分析结果结构化保存，以及登录鉴权和演示脚本。

### 18. 如果老师问“创新点是什么”？

可以回答：

我会从三个方面回答：第一，实时行情进入数据库并参与回测；第二，回测任务采用异步 job 链路，便于状态管理和失败重试；第三，模型分析基于结构化回测指标输出固定字段，并保留 provider 与 fallback 信息。
