# 基于实时行情与大语言模型的量化回测平台设计与实现

## 摘要

本项目设计并实现了一套面向虚拟货币现货研究的工程化量化回测平台。系统采用 Vue 3、FastAPI、SQLAlchemy、SQLite 与 Python 量化引擎构建，覆盖用户认证、实时 K 线采集与存储、策略管理、异步回测、风险指标分析和模型辅助分析等功能。平台从 Binance 公开接口增量同步 BTCUSDT、ETHUSDT 多周期 K 线，将数据持久化并检查新鲜度、断档和缺失数量；回测引擎支持均线交叉和 RSI 反转等策略，输出收益率、最大回撤、夏普比率、胜率、交易记录与权益曲线；分析模块基于回测结构化指标生成风险等级、优势、警告、适配场景、执行计划和建议参数，并保留 Gemini、OpenAI 与本地规则引擎切换能力。

截至 2026 年 6 月 15 日，本地演示数据库包含 33 条回测记录、10 条分析记录和 3992 根 K 线，最新 K 线时间为 2026-06-15 02:56。后端 16 项自动化测试全部通过，前端生产构建成功。项目已经形成“实时行情 -> 策略回测 -> 指标评估 -> 分析建议 -> 参数迭代”的完整课程演示闭环。

**关键词：** 量化回测；实时 K 线；虚拟货币；FastAPI；Vue 3；Google Gemini；工程化

## 1. 项目背景与目标

量化交易研究通常涉及行情获取、策略开发、历史回测、指标分析和结果解释。传统脚本式研究容易出现数据来源分散、参数难以复现、结果缺少持久化、非技术用户难以理解等问题。本项目阶段目标是完成一套边界清晰、可重复运行、适合课程展示和后续扩展的量化研究平台。

项目目标包括：

1. 建立独立注册、登录和用户数据隔离能力。
2. 支持虚拟货币多周期实时 K 线拉取、存储和观察。
3. 支持策略参数配置、异步回测和历史结果查询。
4. 以图表方式展示 K 线、成交量、交易标记和权益变化。
5. 接入外部模型能力，对回测结果生成结构化分析建议。
6. 通过配置、数据库迁移、测试和启动脚本体现工程化能力。

课程结项范围限定为虚拟货币现货只做多研究。期货双向、美股和真实下单不属于本阶段交付内容。

## 2. 需求分析

### 2.1 功能需求

- 用户注册、登录、退出和访问令牌校验。
- BTCUSDT、ETHUSDT 行情标的选择。
- 1m、5m、15m、1h、1d 等 K 线周期选择。
- Binance 行情增量同步与 SQLite 持久化。
- 行情新鲜度、缺口和缺失 K 线检查。
- 均线交叉、RSI 反转和自定义策略管理。
- 回测区间、初始资金和策略参数配置。
- 异步任务状态、取消、失败重试和历史查询。
- 收益率、年化收益、最大回撤、夏普比率、胜率等指标展示。
- Gemini、OpenAI 和本地引擎三种 AI 分析模式。
- AI 结果持久化和历史分析查询。

### 2.2 非功能需求

- 可复现：网络异常时可使用数据库缓存和 CSV 样例。
- 可维护：前端、后端和量化引擎职责分离。
- 可测试：核心服务和量化计算具备自动化测试。
- 可演示：一条命令启动，固定端口并提供自检脚本。
- 可扩展：数据源、模型 provider 和策略实现均保留扩展边界。

## 3. 系统总体设计

系统采用前后端分离架构，量化引擎作为独立 Python 包由后端调用。

```text
用户浏览器
    -> Vue 3 + Vite + ECharts
    -> FastAPI API / 鉴权 / 业务编排
    -> SQLAlchemy + SQLite
    -> Quant Engine 回测与指标计算
    -> Binance REST 行情数据
    -> Google Gemini / OpenAI / Local Engine
```

前端负责登录、参数输入、任务状态和图表展示；后端负责权限、数据同步、任务编排、持久化和外部模型调用；量化引擎负责信号、撮合、费用、权益和指标计算。该分层避免将策略计算散落在页面或路由中。

## 4. 核心模块实现

### 4.1 用户认证与数据隔离

系统使用 PBKDF2 保存密码摘要，通过 HMAC 签名访问令牌识别当前用户。策略、回测记录和 AI 分析记录都带有 `user_id`。未登录用户只能看到独立登录/注册页，演示启动默认关闭匿名开发回退。

### 4.2 实时行情采集与质量观察

后端通过 Binance REST K 线接口按标的和周期增量同步数据，并对 `(symbol, timeframe, open_time)` 进行幂等写入。前端行情页每 10 秒刷新，可显示 K 线、成交量、MA7 和 MA25。

行情接口还返回健康信息：

- `fresh`：数据处于预期更新范围。
- `watch`：数据有轻微延迟，需要观察。
- `stale`：数据明显过期。
- `empty`：没有可用数据。
- `gap_count` 与 `missing_count`：用于识别时间序列断档。

回测服务优先使用数据库中的近期行情；若请求区间覆盖不足，则回退到版本库中的 CSV 样例。这一设计兼顾实时演示和离线可复现性。

### 4.3 量化回测引擎

回测引擎按 K 线顺序推进，策略在当前完整 K 线结束后产生信号，订单最早在下一根 K 线执行，从而避免同 K 线前视偏差。Broker 统一管理现金、持仓、手续费、交易记录和权益曲线。

当前内置策略包括：

- 均线交叉策略：短周期均线上穿长周期均线买入，下穿卖出。
- RSI 反转策略：根据超买、超卖阈值生成交易信号。

性能指标包括总收益率、年化收益率、最大回撤、夏普比率、胜率、交易次数和最终权益。不同周期使用对应年化系数，避免将小时级收益按日线错误年化。

### 4.4 异步回测任务

前端提交回测后，后端创建 `queued` 状态任务，并由进程内线程池执行。任务状态包括 `queued`、`running`、`succeeded`、`failed` 和 `cancelled`。任务中心支持轮询、取消、失败重试和打开成功结果。

该实现适合课程演示和单实例部署。生产环境应进一步替换为独立消息队列、Worker 和心跳恢复机制。

### 4.5 智能推荐分析

智能分析模块基于回测结果组织结构化上下文，包括策略名称、标的、周期、回测区间、初始资金、策略参数、收益率、最大回撤、夏普比率、胜率和交易次数。

模型返回统一 JSON 结构：

- 总结、风险等级和策略类型。
- 推荐评分与置信度。
- 优势、警告和行动建议。
- 适配场景与不建议场景。
- 下一步执行计划和建议参数。

默认 provider 为 Google Gemini，模型为 `gemini-3.5-flash`。系统同时保留 OpenAI Responses API 和本地规则引擎。外部 API 失败时会明确显示失败原因并回退本地分析，页面会保留实际 provider 信息。

## 5. 实际运行案例

2026 年 6 月 15 日对 BTCUSDT 日线均线交叉策略进行回测，区间为 2025-10-12 至 2026-06-15，初始资金为 10000。结果如下：

| 指标 | 结果 |
|---|---:|
| 总收益率 | -7.15% |
| 最大回撤 | -16.75% |
| 夏普比率 | -0.399 |
| 胜率 | 25% |
| 交易次数 | 8 |
| 最终权益 | 9285.45 |

Gemini 对该结果给出 35/100 的推荐评分和 85% 的置信度，识别出低胜率、负夏普和震荡行情假信号问题，并建议加入 ADX、ATR 或成交量过滤，增加止损机制，并对均线周期进行网格搜索。该案例用于展示模型分析如何结合回测指标给出风险判断和后续参数实验方向。

## 6. 工程化设计

### 6.1 配置与密钥管理

数据库、CORS、认证、行情源和 AI provider 均通过环境变量配置。真实密钥放在被 Git 忽略的 `backend/.env.local`，仓库只提供 `.env.example`。

### 6.2 数据库迁移

项目使用 SQLAlchemy 建模，并通过 Alembic 维护正式 schema。开发环境为兼容旧 SQLite 数据库保留自动补列逻辑，生产环境应只使用迁移脚本。

### 6.3 自动化测试

当前 16 项后端测试覆盖：

- 用户注册、登录、令牌解析和未授权访问。
- 策略接口用户隔离。
- 回测指标稳定性和空区间错误。
- 回测记录持久化。
- 异步任务执行、取消、重试和恢复。
- 实时 K 线写入、排序、日期转换和断档检查。
- AI provider 模式切换、结果持久化和额度异常回退。

前端通过 Vite 生产构建验证依赖和组件编译。

### 6.4 演示运维

项目提供以下脚本：

- `scripts/demo.sh`：推荐的一键演示入口，固定启动 8000/5173。
- `scripts/check_demo.sh`：检查服务、数据库、K 线和 Gemini 配置。
- `scripts/start_demo.sh` / `stop_demo.sh`：后台启动与停止。

## 7. 项目特点与创新

1. **形成完整数据闭环。** 实时行情经过持久化和质量检查后参与回测。
2. **模型分析与量化指标结合。** 分析输入来自结构化回测结果，输出遵循固定字段并保存历史。
3. **provider 可切换且可追溯。** 页面明确展示 Gemini、OpenAI 或本地引擎，不混淆来源。
4. **负面结果同样可解释。** 系统不追求展示虚假高收益，而是强调风险识别和参数迭代。
5. **课程项目具备工程化基础。** 包含认证、迁移、异步任务、测试、配置和演示脚本。

## 8. 局限性与后续工作

- 自定义 Python 策略仍运行在 API 进程内，需要子进程沙箱、超时和资源限制。
- 异步任务使用进程内线程池，需要独立队列和 Worker 才适合多实例部署。
- 行情定时同步目前由浏览器触发，后续应迁移到服务端调度器。
- 策略建议尚未自动进行样本外候选回测和参数排名。
- 当前只支持虚拟货币现货只做多，不包含杠杆、做空、资金费率和实盘下单。
- 生产环境还需 HTTPS、标准 JWT/OAuth、结构化日志、请求 ID、CI 和端到端测试。

## 9. 总结

本项目完成了从实时行情获取、数据存储、策略回测、风险指标到模型辅助分析的完整研究流程，并通过用户隔离、异步任务、数据库迁移、自动化测试和一键演示脚本提升了可维护性与可展示性。项目当前已经满足课程结项演示的核心要求，同时明确区分课程演示能力与生产级交易系统要求，为后续扩展策略沙箱、纸面交易和独立任务服务保留了清晰路径。
